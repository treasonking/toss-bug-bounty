# 취약점 유형별 PoC 플레이북 (안전 검증용)

## 공통 운영 규칙
- 모든 요청은 고정 `User-Agent credential` 적용
- 자동화 대량 요청 금지, 기능당 단건/소수 요청만 수행
- 실제 피해 유발 금지, 가능성 입증 후 즉시 중단

## 1) 인증/인가 우회
### 목표
- 권한 없는 사용자가 보호 자원 조회/수정 가능한지 확인

### 최소 PoC 절차
1. 계정 A로 정상 요청 캡처
2. 계정 B 세션으로 동일 요청 재현
3. 식별자 1개만 바꿔 단건 재전송

### 캡처 포인트
- 정상/비정상 요청 Raw
- 응답 코드 비교(200 vs 403/404)
- 핵심 필드 노출 여부

## 2) IDOR/BOLA
### 목표
- 객체 식별자 치환만으로 타인 자원 접근 가능한지 확인

### 최소 PoC 절차
1. A가 생성한 리소스의 `id` 확보
2. B 세션으로 `id`만 치환해 요청
3. 응답 본문 길이/핵심 값 비교

### 캡처 포인트
- `userId`, `accountId`, `transactionId`, `fileId` 변경 전후
- 권한 오류 처리 일관성(403/404 정책)

## 3) 파일 업로드/다운로드 접근통제
### 목표
- 파일 소유권 검증이 서버에서 강제되는지 확인

### 최소 PoC 절차
1. A로 파일 업로드 후 식별자/다운로드 URL 확보
2. B 세션으로 동일 파일 접근 시도
3. 만료 후 URL 재사용 가능 여부 단건 확인

### 캡처 포인트
- 업로드 응답의 `fileId`/`ownerId`
- 다운로드 시 권한 오류 유무

## 4) SSRF 가능성
### 목표
- 서버가 사용자 입력 URL로 요청을 수행하는지 확인

### 최소 PoC 절차
1. URL 입력 기능에서 `http://bugbounty.toss.sb` 제출
2. 서버 처리 결과/콜백 발생 여부 확인
3. 우회 시도 없이 단일 케이스로 입증 종료

### 캡처 포인트
- URL 제출 요청
- 서버 응답 및 검증용 도메인 접속 증적

## 5) 금액변조
### 목표
- 금액/수수료가 서버에서 재검증되는지 확인

### 최소 PoC 절차
1. 정상 100원 요청 캡처
2. `amount`/`fee` 단건 변조 요청
3. 서버가 차단 또는 재계산하는지 확인

### 캡처 포인트
- 변조 전후 요청 바디
- 서버 계산 결과 차이

## 6) XSS/CSRF
### 목표
- 입력값 필터링/출력 인코딩/요청 위조 방어 확인

### 최소 PoC 절차
1. 문의/프로필 입력에 무해 테스트 페이로드 입력
2. 저장/조회 시 반사·저장 여부 확인
3. 상태변경 요청에 CSRF 방어토큰 요구 여부 확인

### 캡처 포인트
- 입력 시점 요청 본문
- 렌더링 결과 화면
- CSRF 관련 헤더/토큰 존재 여부

## 7) Misconfiguration
### 목표
- 보안 헤더, 캐시, CORS, 오류 응답 오노출 점검

### 최소 PoC 절차
1. 핵심 API 3개 선택
2. 응답 헤더/오류 메시지 비교
3. 디버그 정보/민감정보 노출 여부 확인

### 캡처 포인트
- `Cache-Control`, `CSP`, `X-Frame-Options`, `Access-Control-*`
- 오류 응답의 stack trace/내부 경로
