# 취약점 제보서(폼 양식 맞춤) - IDOR/BOLA 샘플

## 취약점 유형
중요 정보 유출이 가능한 취약점

## 대상
https://tossbank.com

## 취약점 제목
프로필 조회 API에서 타 사용자 정보 조회 가능 (IDOR/BOLA)

## 공격 환경
Android 앱 vX.Y.Z / 계정 A,B / Burp Repeater / UA credential 고정 / 2026-04-09 11:40 KST

## 공격 페이로드
`GET /api/v1/profile/A12345` 를 계정 B 세션으로 호출

## 취약점 설명 및 발생원인
프로필 조회 API(`GET /api/v1/profile/{userId}`)에서 요청 주체와 객체 소유자 매핑 검증이 약해 보이는 정황이 확인되었습니다.  
계정 A로 확보한 `userId`를 계정 B 세션 요청 경로에 넣어 호출했을 때, 차단(403/404) 대신 200 응답과 함께 일부 프로필 필드가 반환되는 케이스가 관찰되었습니다(샘플 시나리오).

재현 절차:
1. 계정 A로 프로필 조회 요청을 보내 `userId=A12345`를 확인합니다.
2. 계정 B 세션으로 동일 엔드포인트를 호출하며 경로 변수만 `A12345`로 지정합니다.
3. 응답 코드와 본문(`nickname`, `maskedPhone`)을 비교합니다.

실제 결과:
- 계정 B 요청임에도 200 응답과 타계정 프로필 필드 일부가 반환됨.

기대 결과:
- 비소유 계정 요청은 403 또는 404로 차단되고 타계정 정보가 반환되지 않아야 함.

발생원인 추정:
- 인증 여부는 확인하지만 객체 수준 권한 검증(요청 사용자 == 리소스 소유자) 강제가 누락된 것으로 추정.

## 취약점 파급력
사용자 프로필 정보 노출 가능성이 있으며, 식별자 예측/유출과 결합되면 다수 사용자 정보 접근으로 확장될 수 있습니다.  
개인정보 노출 이슈로 서비스 신뢰도 저하 및 규제 리스크가 발생할 수 있습니다.

## 취약점 대응방안
모든 객체 조회 API에서 객체 수준 권한 검증을 강제하고, 권한 실패 응답을 일관화(403/404)해야 합니다.  
또한 프로필/거래/파일 등 식별자 기반 API에 대해 서버단 권한 테스트 케이스를 CI에 추가해 재발을 방지해야 합니다.
