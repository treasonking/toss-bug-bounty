# 샘플 제보서: IDOR/BOLA

## 제목
[IDOR/BOLA] 프로필 조회 API에서 타 사용자 정보 조회 가능

## 취약점 유형
- IDOR/BOLA (Broken Object Level Authorization)

## 대상 URL/API
- Method: `GET`
- Endpoint: `/api/v1/profile/{userId}`
- 앱/웹 버전: `모바일 앱 vX.Y.Z`

## 공격 환경
- 테스트 일시: `2026-04-09 10:20 KST`
- 테스트 계정: A(소유자), B(비소유자)
- User-Agent(credential 고정): 적용
- 프록시/Burp 사용 여부: 사용

## 전제조건
- 계정 A/B 로그인 세션 필요
- A 계정 `userId` 확보 가능

## 재현 절차
1. 계정 A로 프로필 조회 요청을 전송해 `userId=A123`를 확인합니다.
2. 계정 B로 로그인한 뒤 동일 API 요청에서 경로 변수만 `A123`으로 치환합니다.
3. 요청 전송 후 응답 본문에 계정 A의 프로필 필드가 포함되는지 확인합니다.

## 실제 결과
- 계정 B 요청임에도 응답 코드 `200`과 함께 계정 A의 닉네임/전화번호 마스킹 값이 반환됨

## 기대 결과
- 계정 B가 소유하지 않은 `userId` 요청은 `403` 또는 `404`로 차단되어야 함

## 파급력
- 타 사용자 개인정보 조회 가능성
- 계정 식별자 열거와 결합 시 다수 사용자 정보 노출 위험

## 발생 원인 추정
- 서버가 인증만 확인하고 리소스 소유권 매핑 검증을 수행하지 않음

## 대응방안
- 모든 객체 조회 시 `요청 사용자 == 리소스 소유자` 검증 강제
- 권한 실패 시 일관된 차단 응답(403/404 정책 통일) 적용
