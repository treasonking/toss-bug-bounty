# 토스 버그바운티 실행 런북 (실무용)

## 0) 전제와 운영 원칙
- 본 문서는 **실제 피해 유발 금지**, **가능성만 안전하게 입증** 원칙을 따른다.
- 모든 요청은 사전 발급 credential을 포함한 고정 `User-Agent`로 수행한다.
- SSRF는 반드시 `http://bugbounty.toss.sb` 도메인으로만 검증한다.
- 금액변조는 반드시 **100원 최소 금액**으로만 검증한다.
- 자동화 대량 스캔/가용성 영향 테스트/무차별 브루트포스는 금지한다.
- 버그바운티 과정에서 취득한 정보/데이터/화면은 외부 공유 금지한다.

## 1) 처음 시작할 때 무엇부터 해야 하는지

### 1-1. 전용 앱 설치 후 초기 환경 세팅
- `테스트 단말 분리`: 개인 실사용 단말이 아닌 테스트 전용 기기/에뮬레이터 사용.
- `계정 분리`: 최소 2개 계정 준비 (`A=공격자`, `B=피해자 시나리오`).
- `시간 동기화`: OS 시간 자동 동기화 켜기 (로그 타임라인 정합성 확보).
- `프록시 신뢰설정`: Burp CA 인증서 설치(모바일/에뮬레이터 포함).
- `네트워크 고정`: 불필요 앱 종료, 백그라운드 트래픽 최소화.

### 1-2. Burp 프로젝트 구성 권장안
- `Project`: `toss-bugbounty-YYYYMMDD` 이름으로 일자 단위 생성.
- `Scope`: 테스트 대상 도메인/API만 포함.
- `Proxy HTTP history`: 폴더 태그 규칙 적용.
  - `AUTH`, `PROFILE`, `PAYMENT`, `UPLOAD`, `URL_INPUT`, `CS`
- `Target site map`: 기능별 폴더 북마크.
- `Repeater 탭`: 기능별로 고정 탭 생성.
  - `01_authn_authz`, `02_idor`, `03_ssrf`, `04_amount`, `05_upload`
- `Logger`: 중요 요청/응답만 별도 마킹(노이즈 제거).

### 1-3. User-Agent credential 고정 방법
- Burp `Proxy -> Match and Replace` 사용.
- 규칙:
  - Type: `Request header`
  - Match: `^User-Agent:.*$`
  - Replace: `User-Agent: <발급받은_credential_포함_고정값>`
- 적용 확인:
  - Repeater 1건 전송 후 Raw 요청에서 UA 값 확인.
  - 앱 내 주요 기능 3개(로그인/조회/수정)에서 동일 UA 유지 확인.
- 운영 수칙:
  - 테스트 중 UA 수동 변경 금지.
  - 브라우저/앱/스크립트 혼용 시에도 동일 UA 강제.

### 1-4. 테스트 로그/스크린샷/PoC 메모 정리 체계
- 디렉터리 규칙:
  - `evidence/YYYY-MM-DD/<기능명>/`
  - `reqres/`, `screens/`, `notes/`
- 파일명 규칙:
  - `HHMM_<기능>_<행위>_<계정>.txt/png`
  - 예: `1410_payment_amount_tamper_A.txt`
- 메모 템플릿(요청마다):
  - 목적 / 전제조건 / 요청ID / 변조 파라미터 / 결과 / 재현성 / 위험도 메모
- 증거 최소 세트:
  - 변조 전후 요청 1쌍, 응답 1쌍, 화면 1장, 타임스탬프 1개

## 2) 초반 1주일 전략 (Day 1 ~ Day 7)

### Day 1: 환경 검증/기준선 수집
- 목표: 테스트 환경 안정화, 정상 플로우 베이스라인 수집.
- 확인 기능: 로그인, 로그아웃, 내 정보 조회, 기본 송금/결제 조회.
- 수집 증거: 정상 요청/응답 샘플, 세션 토큰 수명, UA 고정 증적.
- 주의사항: 기능 탐색만, 변조 최소화.

### Day 2: 인증/인가 경계 확인
- 목표: 계정 A/B 간 리소스 분리 정책 파악.
- 확인 기능: 프로필 조회/수정, 거래내역 조회 API.
- 수집 증거: 동일 엔드포인트의 A/B 비교 요청 세트.
- 주의사항: 타 계정 데이터 노출 시 즉시 중단 후 증거 보존.

### Day 3: IDOR/BOLA 집중
- 목표: 객체 식별자(`userId`, `accountId`, `transactionId`) 변조 검증.
- 확인 기능: 상세조회, 수정, 삭제, 다운로드 API.
- 수집 증거: 정상 ID vs 변조 ID 응답 차이표.
- 주의사항: 대량 ID 열거 금지(소수 샘플만).

### Day 4: 파일 업/다운로드 접근통제
- 목표: 파일 소유권 검증 누락 여부 확인.
- 확인 기능: 영수증/첨부 업로드, 다운로드 링크 접근.
- 수집 증거: 파일 식별자 변경 전후 결과, 권한 오류 코드 여부.
- 주의사항: 악성 파일 업로드 시도 금지, 안전 확장자만.

### Day 5: SSRF 가능성 안전 검증
- 목표: URL 입력/미리보기/웹훅 계열에서 서버측 요청 유도 여부 확인.
- 확인 기능: 외부 URL 등록/검증/미리보기.
- 수집 증거: `http://bugbounty.toss.sb` 요청 유도 성공 증적.
- 주의사항: 내부망/메타데이터 주소 접근 시도 금지.

### Day 6: 금액변조(100원만) + CSRF/XSS 1차
- 목표: 금액/수량/수수료 파라미터 신뢰성 확인.
- 확인 기능: 송금/결제 생성 및 확인 단계 API.
- 수집 증거: 100원 기준 정상/변조 비교 요청.
- 주의사항: 실제 금전 피해 유발 동작 금지, 소액 테스트만.

### Day 7: Misconfiguration + 리포트 정리
- 목표: 헤더/캐시/CORS/오류메시지 오노출 점검 및 제보 초안 완성.
- 확인 기능: 공통 API 응답 헤더, 에러 응답, 권한 실패 처리.
- 수집 증거: 재현 가능한 핵심 취약점 1~2건 완결 패키지.
- 주의사항: 중복/불확실 이슈 제출 금지, 재현성 우선.

## 3) 우선적으로 볼 취약점 유형

### 3-1. 인증/인가 우회
- 왜 먼저: 영향도 높고 재현 증거가 명확함.
- 주로 찾는 기능: 프로필/결제/거래내역 조회·수정 API.
- 안전 검증: A/B 계정 교차 요청, 토큰 유지한 채 객체만 교체.

### 3-2. IDOR/BOLA
- 왜 먼저: 모바일/API 서비스에서 빈도가 높음.
- 주로 찾는 기능: 상세조회, 파일다운로드, 상태변경 API.
- 안전 검증: 단건 식별자 치환 후 응답코드/본문 비교.

### 3-3. 파일 업로드/다운로드 접근통제
- 왜 먼저: 민감정보 노출 위험 큼.
- 주로 찾는 기능: 첨부파일, 영수증, 프로필 이미지.
- 안전 검증: 소유권 다른 파일ID 대입, 서명 URL 재사용 테스트.

### 3-4. SSRF 가능성
- 왜 먼저: 내부 리소스 접근으로 확장될 가능성.
- 주로 찾는 기능: URL 입력, 링크 미리보기, 콜백/웹훅.
- 안전 검증: `bugbounty.toss.sb` 단일 도메인으로만 외부 호출 입증.

### 3-5. 금액변조
- 왜 먼저: 직접적인 비즈니스 임팩트와 직결.
- 주로 찾는 기능: 결제 생성/승인, 송금 금액 파라미터.
- 안전 검증: 100원 트랜잭션 내 파라미터 변조, 서버 계산 검증.

### 3-6. XSS / CSRF
- 왜 먼저: 사용자 세션/행위 위임 리스크.
- 주로 찾는 기능: 고객센터, 프로필 입력, 알림/메시지 영역.
- 안전 검증: 무해 페이로드 중심, 상태변경 요청의 CSRF 방어 확인.

### 3-7. Misconfiguration
- 왜 먼저: 짧은 시간에 다수 표면 점검 가능.
- 주로 찾는 기능: 공통 게이트웨이 응답/에러 페이지/정적 파일.
- 안전 검증: 보안 헤더, CORS, 캐시 정책, 디버그 정보 노출 점검.

## 4) 실제 테스트 체크리스트

### 4-1. 로그인/회원가입
- 캡처: 로그인 요청/응답, 토큰 발급/갱신, 실패 케이스 응답.
- 주시 파라미터: `email`, `phone`, `otp`, `deviceId`, `session`.
- 이상 징후: 비정상 인증 우회, 계정 존재 유추 가능한 상세 에러.

### 4-2. 계정 정보 수정
- 캡처: 수정 전/후 프로필 조회 결과, 수정 API 요청 본문.
- 주시 파라미터: `userId`, `nickname`, `phone`, `email`, `profileImageId`.
- 이상 징후: 타 계정 정보 수정 성공, 서버측 검증 누락.

### 4-3. 첨부파일 업로드/다운로드
- 캡처: 업로드 요청, 파일ID/URL 발급, 다운로드 요청.
- 주시 파라미터: `fileId`, `ownerId`, `contentType`, `path`, `signedUrl`.
- 이상 징후: 권한 없는 파일 접근, 만료 URL 재사용 가능.

### 4-4. 외부 URL 입력 기능
- 캡처: URL 제출 요청, 서버 처리 결과, 외부 호출 증적.
- 주시 파라미터: `url`, `callback`, `previewUrl`, `target`.
- 이상 징후: 서버가 임의 URL 요청 수행, 필터 우회 흔적.

### 4-5. 결제/송금/금액 관련 요청
- 캡처: 금액 생성/확정 단계 각각의 요청/응답.
- 주시 파라미터: `amount`, `fee`, `currency`, `recipientId`, `orderId`.
- 이상 징후: 클라이언트 금액 신뢰, 서버 재계산 미수행.

### 4-6. 고객센터/문의/알림/초대 기능
- 캡처: 문의 등록/조회, 알림 읽음/삭제, 초대 링크 생성 요청.
- 주시 파라미터: `ticketId`, `message`, `inviteCode`, `notificationId`.
- 이상 징후: 타 사용자 문의/알림 접근, 스크립트 반사/저장 가능.

## 6) Burp 기준 실전 팁

### 6-1. Repeater 비교 순서
1. 정상 요청(기준선) 저장
2. 식별자만 변경 (`userId`, `fileId`, `transactionId`)
3. 권한 관련 헤더/쿠키 유지한 채 재전송
4. 응답 코드, 길이, 핵심 필드 차이 비교
5. 차이 발생 시 최소 재현 세트로 축소

### 6-2. 계정 2개 비교 검증 방식
- A 계정으로 리소스 생성 -> 식별자 확보
- B 계정 세션으로 동일 리소스 접근/수정 시도
- 성공/실패 여부와 오류코드(403/404/200) 비교 기록
- 항상 단건 검증 후 중단, 대량 시도 금지

### 6-3. 중점 확인 항목 (Request/Response)
- 헤더: `Authorization`, `Cookie`, `User-Agent`, `Origin`, `Referer`
- 토큰/식별자: `accessToken`, `refreshToken`, `userId`, `accountId`, `fileId`, `orderId`
- 응답: 권한 오류 코드 일관성, 민감정보 마스킹, 디버그 필드 노출 여부

### 6-4. 과도한 요청 없이 효율적으로 검증
- 기능당 핵심 요청 1~3개만 선정해 수동 집중 분석
- 동일 요청의 파라미터 1개씩만 변경(원인 분리)
- 성공 조건이 보이면 즉시 재현성 검증 후 증거화
- 불확실한 가설은 접고 재현 가능한 이슈부터 보고서화

## 가장 먼저 시도할 3가지
1. Burp에서 `User-Agent credential` 고정 규칙 먼저 적용하고 정상 동작 3건 캡처
2. 계정 A/B 준비 후 `IDOR/BOLA` 단건 교차 검증부터 수행
3. URL 입력 기능에서 `bugbounty.toss.sb`로만 SSRF 가능성 안전 입증
